# BUG-2026-06-22T192003: CI — pnpm minimumReleaseAge rejecting astro@7.0.0

## Problem

All 6 CI jobs (typecheck, format, unit, integration, build, e2e) fail at `pnpm install` with:

```
[ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION] 2 lockfile entries failed verification:
  @astrojs/markdown-satteri@0.3.1 was published at 2026-06-22T10:10:10.000Z, within the minimumReleaseAge cutoff
  astro@7.0.0 was published at 2026-06-22T10:10:04.000Z, within the minimumReleaseAge cutoff
```

**Actual behavior:** CI pipeline fails immediately at dependency installation, preventing any job from running.

**Expected behavior:** CI should install dependencies from the committed lockfile without rejecting valid packages.

**How to reproduce:** Push any commit to a branch with the current pnpm-lock.yaml (which pins `astro@7.0.0` and `@astrojs/markdown-satteri@0.3.1`). The CI workflow runs `pnpm install` (without `--frozen-lockfile`), but pnpm v11's supply-chain policy rejects these two packages because they were published very recently (same day as the CI run).

## Root Cause Analysis

Pnpm v11 introduced a supply-chain security feature: `minimumReleaseAge` in lockfile v9. This check rejects packages published within a recent window to protect against dependency confusion or compromised freshly-published packages.

The project's `pnpm-lockfile.yaml` (v9) pins `astro@7.0.0` and `@astrojs/markdown-satteri@0.3.1` — both were published on 2026-06-22 (today), making them younger than the `minimumReleaseAge` cutoff applied by pnpm v11 in the CI environment.

The project has no explicit configuration for `minimumReleaseAge`:

- No `.npmrc` file exists
- No `pnpm.minimumReleaseAge` in `package.json`
- `pnpm-workspace.yaml` only configures `allowBuilds` for esbuild/sharp

Locally, `pnpm install` succeeds because the local pnpm version or configuration doesn't enforce this check as strictly, or the lockfile was generated before the policy took effect.

**Code path involved:**

- CI workflow (`.github/workflows/ci.yml`) → `pnpm install` step
- pnpm v11 lockfile verification → `ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION`
- Rejects `astro@7.0.0` and `@astrojs/markdown-satteri@0.3.1`

**Contributing factors:**

- `astro@7.0.0` was released on the same day as the CI run
- pnpm v11's default policy considers 0-day-old packages risky
- No project-level configuration overrides the default

**Risk level:** Low — does not affect local development, only CI. Fix is a one-line config addition.

## Resolution

**Status:** RESOLVED (2026-06-22)

**Actual fix applied (3 changes):**

1. **Pnpm version downgrade (`.github/workflows/ci.yml`):** Changed `PNPM_VERSION` from `'11'` to `'10'`. Pnpm v11 enforces a `minimumReleaseAge` supply-chain policy on lockfile v9 that rejects packages published within ~24 hours. Both `astro@7.0.0` and `@astrojs/markdown-satteri@0.3.1` were published on 2026-06-22. Pnpm v10 does not have this policy.

2. **Vitest project filter (`package.json`):** Both `test:unit` and `test:integration` scripts used `vitest run --project <name>`, but no vitest projects were configured. Changed to plain `vitest run` which picks up the default config.

3. **Playwright configuration and test:** Created `playwright.config.ts` with `testDir: './e2e'` to isolate e2e tests from vitest tests. Created placeholder e2e test. Added `@playwright/test` as devDependency.

**Verification:** CI run `27979474324` — all 6 jobs passed (unit, typecheck, format, e2e, build, integration).
